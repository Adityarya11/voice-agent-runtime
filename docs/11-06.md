# Voice Agent Runtime — Project Summary

## What this project is

A distributed, real-time voice agent runtime built from scratch on a
mid-range consumer laptop. The system enables multiple domain-specific
AI calling agents to be configured through simple YAML profiles and
run through a shared inference engine — without any architectural changes
to the runtime itself.

The same engine that powers a dental receptionist can power a real estate
consultant or a college admissions counselor. Only the configuration changes.
The runtime does not know or care about the domain.

---

## Technology decisions and why

**Go for orchestration.**
Go was chosen for the orchestrator because goroutines and channels are
first-class primitives. Concurrent session handling — where each active
call gets its own isolated goroutine pool — is idiomatic Go and has near-zero
overhead per session. The alternative, threading in Python, would have been
heavier and harder to reason about at scale.

**Python for inference.**
The entire AI ecosystem — Faster-Whisper, Ollama, Piper — has Python as its
primary interface. Putting inference in Python and keeping it there means
we never fight the tooling.

**gRPC with a unified Event message.**
A single bidirectional stream carrying a `oneof` Event payload means the
protocol can evolve — adding transcripts, tool calls, memory updates — without
breaking the transport contract. This was a deliberate early decision and it
has already proven correct as the system grew.

**Hardware-aware component placement.**
RTX 3050 Mobile with 4GB VRAM and a 6-core Ryzen CPU drives every placement
decision. LLM on GPU (~2.2GB VRAM), STT and TTS on CPU. Nothing fights for
the GPU. Nothing OOMs. This is a real constraint that was solved properly
rather than ignored.

## What has been built so far

### Phase 1 — gRPC bridge

Established bidirectional streaming between the Go orchestrator and the
Python inference engine over gRPC. Proved the transport layer works end
to end with real audio.

### Phase 2 — Inference pipeline

Integrated Faster-Whisper for speech to text, Ollama running qwen2.5:3b
for language model inference, and Piper TTS for synthesis. A complete
STT → LLM → TTS pipeline running locally with no external API dependencies.

### Phase 3 — Dynamic profiling

The system can load any agent profile YAML at startup. The profile's system
prompt flows from Go through the proto contract into Python and reaches the
LLM. Verified: the model adopts the loaded persona with confirmed GPU
utilization. The runtime is genuinely domain-agnostic.

### Phase 4 — Session management

The Session struct was promoted from a passive data container to an active
controller. It now owns the gRPC stream, enforces state transitions through
a validated transition map protected by a mutex, and runs writePump and
readPump goroutines that own all gRPC traffic. main.go has no knowledge
of gRPC internals — it only interacts with channels.

The state machine enforces this lifecycle:

```text
CREATED → CONNECTING → ACTIVE → PROCESSING → RESPONDING → TERMINATED
```

Illegal transitions are caught and logged rather than silently corrupting
state or panicking.

## What is coming next

**Token streaming with sentence-boundary chunking.**
The LLM currently waits until the full response is generated before TTS
begins. The next milestone pipes tokens out of the LLM as they arrive,
accumulates them to sentence boundaries, and feeds each sentence to TTS
immediately. Sentence one plays while sentence two is still being generated.
This collapses the largest latency gap in the pipeline.

**True bidirectional duplex.**
The stream is currently half-duplex — Go sends everything, closes send,
waits. True duplex means VAD on the Python side detects end of speech and
begins inference without waiting for CloseSend, while Go simultaneously
receives audio. This makes the system behave like a real phone call.

## Known architectural debt being tracked

- Monitor goroutine for context cancellation, timeout, and barge-in support.
- sync.Once fix for DoneChan to safely handle multiple goroutines signaling
  session completion without panic.
- Temp file boundary in Python STT (entire audio buffered to disk before
  transcription begins — acceptable for V1, incompatible with true streaming).
- VAD integration for live speech boundary detection in the audio stream.
